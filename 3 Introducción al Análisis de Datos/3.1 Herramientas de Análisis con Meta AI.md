## Evolución: Explica la relación técnica entre NumPy y Pandas. ¿Por qué se dice que Pandas es una evolución de NumPy para el análisis de datos tabulares?

**NumPy** y **Pandas** no compiten: Pandas está construido *encima* de NumPy. Literalmente es una capa de abstracción que usa NumPy por debajo.

### Relación técnica: Pandas usa NumPy como motor

| Aspecto | NumPy | Pandas | Relación |
| --- | --- |
| **Estructura base** | `ndarray`: array N-dimensional homogéneo | `Series` y `DataFrame` | Un `Series` es un `ndarray` + índice. Un `DataFrame` es un dict de `Series` que comparten índice |
| **Tipo de datos** | Un solo `dtype` por array | Múltiples `dtypes` por columna | Cada columna de un DataFrame es un `ndarray` de NumPy con su propio tipo |
| **Almacenamiento** | Bloques contiguos en memoria C | Bloques contiguos por columna | Pandas almacena cada columna como un `ndarray`, por eso las operaciones vectorizadas son igual de rápidas |
| **Operaciones** | Matemáticas, álgebra lineal, broadcasting | Manipulación, limpieza, agrupación, joins | Pandas delega los cálculos numéricos a NumPy/UFuncs |

Puedes comprobarlo así:
```python
import pandas as pd
import numpy as np

df = pd.DataFrame({'a': [1, 2, 3]})
type(df['a'].values)  # <class 'numpy.ndarray'>
```
El `.values` te regresa el `ndarray` de NumPy que vive adentro.

### ¿Por qué Pandas es una "evolución" para datos tabulares?

NumPy es brutalmente rápido para números, pero fue diseñado para álgebra lineal y computación científica. Cuando trabajas con datos tipo Excel/CSV te faltan cosas:

1. **Etiquetas en los ejes** 
   NumPy: `arr[0, 2]` - tienes que recordar que la fila 0 es "usuario_X" y la columna 2 es "edad". 
   Pandas: `df.loc['usuario_X', 'edad']`. Índices con nombre para filas y columnas. Esto elimina bugs y hace el código legible.

2. **Heterogeneidad de datos**
   Un `ndarray` solo guarda un tipo: todo `float64` o todo `str`. Si mezclas, NumPy hace cast a `object` y pierdes velocidad.
   Pandas maneja un `dtype` por columna: `int64`, `float64`, `datetime64`, `string`, `category`. Por eso un DataFrame puede tener edades, nombres y fechas sin problema.

3. **Datos faltantes** 
   NumPy original solo tenía `np.nan` para floats. Si tu array era de `int`, no podías tener NaNs.
   Pandas introdujo `pd.NA`, `pd.NaT` y manejo robusto de nulos con `.isna()`, `.fillna()`, `.dropna()`. Clave para datos del mundo real.

4. **Operaciones de alto nivel para tablas**
   NumPy no sabe qué es un "join". Pandas trae `.merge()`, `.join()`, `.groupby()`, `.pivot_table()`, `resample()` para series de tiempo. Son operaciones que escribirías en 50 líneas de NumPy y aquí son 1 línea.

5. **Alineación automática de datos**
   Si sumas dos `Series` con índices diferentes, Pandas alinea por etiqueta automáticamente. NumPy solo alinea por posición y si no coinciden truena. Para análisis financiero/temporal esto es oro.

### La analogía rápida

**NumPy es el motor V8**: potencia bruta, eficiente, bajo nivel.  
**Pandas es el auto completo**: usa ese motor V8, pero ya trae volante, GPS, aire acondicionado y porta vasos. No quieres construir un auto desde el motor cada vez que analizas un CSV.

Por eso decimos que es una evolución: toma toda la velocidad de C que te da NumPy y le pone encima las abstracciones que necesitas para trabajar con datos tabulares sucios, etiquetados y heterogéneos.

## Objetos: Dame un ejemplo de la vida real (fuera de la programación) que ilustre la diferencia entre una Series y un DataFrame de Pandas.

Piensa en un **expediente escolar en papel**. Ahí tienes la diferencia perfecta.

### Series = Una sola columna del expediente

Imagina la **hoja de calificaciones de Matemáticas** de todo el salón.

| Alumno | Calificación Mate |
| --- | --- |
| Ana | 9.5 |
| Luis | 8.0 |
| Sofía | 10.0 |
| Diego | 7.8 |

Eso es una `Series`. 
- **Tiene un solo tipo de dato**: todas son calificaciones.
- **Tiene etiquetas**: el nombre de cada alumno es el índice.
- **Responde una sola pregunta**: "¿Qué sacó cada alumno en Matemáticas?"

Si quieres solo esa hoja, no necesitas el expediente completo. Es un objeto 1D: una lista de valores + sus etiquetas.

### DataFrame = El expediente completo del salón

Ahora junta todas las hojas del expediente:

| Alumno | Mate | Historia | Ciencias | Asistencia |
| --- | --- | --- | --- | --- |
| Ana | 9.5 | 10.0 | 9.0 | 98% |
| Luis | 8.0 | 7.5 | 8.5 | 92% |
| Sofía | 10.0 | 9.8 | 10.0 | 100% |
| Diego | 7.8 | 8.0 | 7.0 | 85% |

Eso es un `DataFrame`.
- **Tiene múltiples tipos de datos**: calificaciones y porcentajes.
- **Cada columna es una Series**: La columna "Mate" es exactamente la Series de arriba.
- **Responde preguntas complejas**: "¿Quién tiene >9 en todo?", "¿Cuál es el promedio de Sofía?", "Correlación entre asistencia y calificaciones".

### La regla mental

**Series** = Una lista con etiquetas. Como la lista de precios de un menú de tacos. Solo "taco" -> "precio".  
**DataFrame** = Una tabla completa. Como el menú completo: "taco" -> "precio", "ingredientes", "calorías", "picor".

Técnicamente: un `DataFrame` es literalmente un diccionario de `Series` que comparten el mismo índice. Por eso si agarras `expediente['Mate']` te regresa una `Series`.

## Comparación: ¿En qué escenarios es superior utilizar Pandas en lugar de Microsoft Excel para el análisis de datos? Menciona al menos 3 ventajas técnicas relacionadas con el volumen de datos y la automatización.

Excel es increíble para explorar rápido y hacer tablas visuales. Pero cuando pasas de "hoja de cálculo" a "análisis de datos serio", Pandas gana por goleada. 

Aquí van **3 ventajas técnicas clave** donde Excel se queda corto:

### 1. Volumen de datos: Pandas no se atora con millones de filas
**Excel**: Límite duro de 1,048,576 filas por hoja. Si tu CSV tiene 10M de registros de ventas, tienes que partirlo, filtrar antes de abrir, o usar Power Query y rezar. Abrir un archivo de 500 MB puede colgar tu máquina.

**Pandas**: El límite es tu RAM. Con 16 GB puedes cargar ~50-100M de filas sin problema. Y si no caben, usas `chunksize` para procesar por partes, o `dask`/`polars` que usan Pandas como API.
```python
# Leer 50M de filas en partes de 1M sin tronar la RAM
for chunk in pd.read_csv('ventas_50M.csv', chunksize=1_000_000):
    procesar(chunk)
```
**Ventaja técnica**: Pandas opera sobre `ndarray` de NumPy en C. Las operaciones vectorizadas no crean celdas individuales con formato como Excel. Por eso `df['precio'] * 1.16` en 10M de filas tarda 0.1s. En Excel, recalcular 1M de fórmulas te vas por un café.

### 2. Automatización: Código reproducible vs clics manuales
**Excel**: Para repetir un análisis el próximo mes tienes que: abrir archivo, filtrar, copiar-pegar, actualizar tablas dinámicas, rehacer gráficas. 20 clics. Si se te olvida un paso, el resultado cambia. Macros VBA ayudan, pero son frágiles y nadie quiere mantenerlas.

**Pandas**: Escribes el análisis una vez en Python y lo corres 1000 veces igual.
```python
# Este script se corre solo cada lunes a las 8am
df = pd.read_csv('ventas.csv')
reporte = df.groupby('region')['ingreso'].sum().sort_values()
reporte.to_excel('reporte_semanal.xlsx')
```
**Ventaja técnica**: Todo es un script versionable en Git. Puedes hacer pruebas unitarias, documentarlo, y correrlo en un servidor sin abrir ninguna interfaz. Excel no tiene `git diff` ni CI/CD. Si tu jefe pide "corre esto con los datos de 2024", cambias 1 línea, no rehaces todo.

### 3. Operaciones complejas: Joins y merges sin volverte loco
**Excel**: Hacer `BUSCARV`/`XLOOKUP` entre 3 tablas de 100k filas cada una es lento y propenso a error. Si una clave no existe te regresa `#N/A` y tienes que envolver todo en `SI.ERROR`. Unir 5 fuentes distintas es un infierno de hojas auxiliares.

**Pandas**: `.merge()` hace joins tipo SQL optimizados en C. Unir 5 tablas de 1M de filas cada una tarda segundos.
```python
clientes = pd.read_csv('clientes.csv')      # 2M filas
pedidos = pd.read_csv('pedidos.csv')        # 10M filas  
productos = pd.read_csv('productos.csv')    # 50k filas

# Left join, inner join, outer join como en SQL
resultado = pedidos.merge(clientes, on='id_cliente').merge(productos, on='sku')
```
**Ventaja técnica**: Pandas usa algoritmos de hash join de bajo nivel. No recalcula toda la hoja cuando cambias una celda. En Excel, cada `BUSCARV` es una función que se reevalúa y te mata el performance con datasets grandes.

### Bonus: Cosas que Excel simplemente no hace bien
- **Series de tiempo**: `df.resample('D').mean()` para pasar datos por minuto a promedios diarios. En Excel es un dolor.
- **Datos no tabulares**: Leer JSON anidado, APIs, o hacer web scraping y meterlo directo a un DataFrame.
- **Integración**: Pasarle tu DataFrame a scikit-learn para ML, o a matplotlib/seaborn para gráficas. Excel vive aislado.

**Regla rápida**: Si lo abres, le echas un ojo y lo cierras -> Excel.  
Si lo vas a procesar, transformar, y repetir cada semana con datos nuevos -> Pandas.
