# Buenas Prácticas del EDA con Meta AI

## 1. Reproducibilidad en Data Science: ¿Por qué la reproducibilidad es considerada uno de los pilares fundamentales de un proyecto de Ciencia de Datos profesional? ¿Cuáles son las principales prácticas para garantizarla en un notebook de Google Colab o Jupyter, y qué consecuencias tiene para un equipo de trabajo cuando un análisis no puede ser reproducido?

La reproducibilidad es el "botón de replay" de Data Science. Si no puedes darle play y obtener el mismo resultado, no es ciencia, es suerte.

### **¿Por qué es uno de los pilares fundamentales?**

**1. Confianza y credibilidad**  
Un modelo que no se puede reproducir es indefendible. Nadie va a poner en producción algo que dio 95% de accuracy el lunes y 60% el martes sin que nadie sepa por qué. Stakeholders, auditores y reguladores exigen poder verificar cada número.

**2. Colaboración real**  
En un equipo, tu notebook de hoy es el bug de alguien mañana. Si tu compañero no puede correr tu análisis y llegar a lo mismo, el trabajo se detiene. Es la diferencia entre "toma, sigue desde aquí" vs "tengo que empezar de cero porque esto no me corre".

**3. Iteración y debugging**  
El 80% de DS es probar hipótesis. Si cambiaste el preprocesamiento y el modelo empeora, necesitas volver al estado anterior exacto. Sin reproducibilidad, cada experimento es irreversible.

**4. Compliance y ética**  
En finanzas, salud, y gobierno, necesitas demostrar cómo llegaste a una decisión. GDPR, auditorías SOX, FDA: todos piden trazabilidad. "Confía en mí bro" no pasa una auditoría.

### **Principales prácticas para garantizarla en Colab/Jupyter**

| Área | Qué hacer | En Colab/Jupyter cómo se ve |
| --- | --- | --- |
| **1. Entorno** | Fija versiones exactas de librerías | `!pip install pandas==2.2 scikit-learn==1.5.1` al inicio. Mejor: `requirements.txt` + `!pip install -r requirements.txt` |
| **2. Semillas** | Fija todos los random seeds | `np.random.seed(42)`, `random.seed(42)`, `torch.manual_seed(42)`, `tf.random.set_seed(42)` |
| **3. Datos** | Versiona el dataset y no hagas cambios manuales | Usa rutas fijas a Drive/S3/GCS. `dvc` o `pandas.read_csv('gs://bucket/data_v3.csv')`. Nunca `df.drop()` sin documentar |
| **4. Orden de ejecución** | Evita notebooks "sucios" | `Kernel > Restart & Run All` antes de compartir. Colab: `Entorno de ejecución > Reiniciar y ejecutar todo` |
| **5. Código vs Output** | Separa lógica de resultados | No hardcodees paths tipo `/content/drive/MyDrive/`. Usa variables: `DATA_PATH = '/content/data'` |
| **6. Dependencias externas** | Documenta APIs y servicios | Si usas una API, fija la versión y guarda la respuesta: `requests.get(url, headers={'api-version': '2024-01'})` |
| **7. Documentación** | Explica el "por qué" | Celdas markdown con: objetivo, supuestos, decisiones de features, y limitaciones |
| **8. Control de versiones** | Git para el notebook | Colab: `Archivo > Guardar copia en GitHub`. Usa `nbdime` o `jupyterlab-git` para diffs limpios |
| **9. Hardware** | Registra el entorno | En la celda 1: `!nvidia-smi`, `!python --version`. GPUs dan resultados distintos a CPU en algunos casos |
| **10. Checkpoints** | Guarda modelos y transformaciones | `joblib.dump(model, 'model_v1.pkl')` y `joblib.dump(scaler, 'scaler_v1.pkl')`. No confíes en re-entrenar |

### **¿Qué pasa cuando un análisis NO es reproducible? Consecuencias para el equipo**

**1. Tiempo quemado = dinero quemado**  
Caso real: Un senior data scientist se va y su modelo en producción falla. Nadie puede replicar cómo se entrenó. El equipo pasa 3 semanas haciendo ingeniería inversa en lugar de entregar features nuevas. 3 semanas × 4 personas = pérdida de 1 sprint completo.

**2. Pérdida de confianza cross-team**  
Producto te pide cambiar un threshold del modelo. Tú dices "no puedo porque si re-entreno cambia todo y no sé por qué". PM deja de confiar en DS. Fin.

**3. Deuda técnica exponencial**  
Cada notebook no reproducible es un "experimento de Schrödinger": funciona hasta que alguien lo toca. En 6 meses tienes 50 notebooks y ninguno sirve como punto de partida. Todo el equipo vive con miedo a `Shift+Enter`.

**4. Riesgo legal y reputacional**  
Banco lanza modelo de crédito no reproducible. Auditor pide explicar por qué rechazó a cliente X. No puedes. Multa + demanda + noticia en El Financiero. 

**5. Imposibilidad de escalar**  
MLOps no existe sin reproducibilidad. No puedes hacer CI/CD, A/B testing, ni monitoring si cada corrida da algo distinto. Te quedas en "notebook-driven development" para siempre.

**Regla de oro**: Si no puedes borrar tu carpeta `/outputs` + darle "Run All" y obtener exactamente los mismos archivos, no es reproducible. 


## 2. Documentación de Decisiones Analíticas: ¿Cuál es la diferencia entre documentar el qué y documentar el por qué en un notebook de análisis de datos? Proporciona un ejemplo concreto de una decisión analítica (como la elección entre media o mediana para imputar) y muestra cómo debería documentarse correctamente tanto en una celda Markdown como en un comentario de código.

**Documentar el *qué* vs el *por qué***

| | Documentar el *QUÉ* | Documentar el *POR QUÉ* |
| --- | --- | --- |
| **Definición** | Describe la acción ejecutada. Es el "hicimos X" | Explica la razón, contexto y trade-offs. Es el "hicimos X *porque* Y" |
| **Pregunta que responde** | ¿Qué código corrió? ¿Qué técnica usaste? | ¿Por qué elegiste esto y no otra cosa? ¿Qué alternativas descartaste? |
| **Vida útil** | Sirve hoy para entender el flujo | Sirve en 6 meses cuando alguien se pregunte si puede cambiarlo |
| **Ejemplo malo** | "Se imputó con la mediana" | *Sin explicación* |
| **Ejemplo bueno** | "Se imputó `edad` con la mediana" | "Se usó mediana porque `edad` tiene outliers >100 años que sesgarían la media, y la mediana preserva la tendencia central robusta" |

**Regla**: El *qué* lo ves leyendo el código. El *por qué* solo existe si lo escribes. Si no documentas el por qué, en 2 meses ni tú sabrás si fue una buena decisión o un hack de último minuto.

### **Ejemplo concreto: Imputar con media vs mediana**

**Contexto**: Tienes una columna `ingreso_mensual` con 8% de valores nulos. El histograma muestra sesgo a la derecha y outliers con ingresos de $500k+.

#### **1. Celda Markdown: Documentación de la decisión**

Así se ve bien documentado antes del código:

```markdown
### 3.2 Imputación de valores nulos en `ingreso_mensual`

**Decisión**: Imputar nulos usando la mediana en lugar de la media.

**Contexto**: La variable `ingreso_mensual` presenta sesgo positivo fuerte, con skewness = 3.2.
El 1% superior tiene ingresos >$250k, mientras que la media es $45k y la mediana $32k.

**Por qué mediana y no media**:
1. **Outliers**: La media de $45k está inflada por valores extremos. Imputar con media asignaría
   $45k a personas con nulos, sobrestimando el ingreso típico.
2. **Impacto en el modelo**: Usamos un modelo lineal sensible a outliers. La mediana es más
   robusta y reduce el riesgo de que los imputados dominen los coeficientes.
3. **Alternativa descartada**: KNN Imputer se probó pero aumentó 40% el tiempo de entrenamiento
   sin mejora en AUC validación. Ver experimento `exp_014_imputacion.ipynb`.

**Supuesto clave**: Asumimos que los nulos son MAR - Missing At Random - y que el ingreso
de los nulos se parece más al "ciudadano típico" que al promedio aritmético.

**Riesgo**: Si los nulos fueran MNAR y correspondieran a personas de altos ingresos que
no reportan, estaríamos subestimando. Validar con equipo de negocio en Q3.
```

Nota el patrón: **Decisión → Contexto → Justificación → Alternativas → Supuestos/Riesgos**.

#### **2. Comentario de código: Documentación inline**

El código sin contexto es peligroso. Así NO:

```python
# Mal: solo el qué
df['ingreso_mensual'].fillna(df['ingreso_mensual'].median(), inplace=True)
```

Así SÍ: *qué* + *por qué* + *link a contexto*

```python
# Imputación de nulos en ingreso_mensual
# POR QUÉ MEDIANA: Distribución con skewness=3.2 y outliers >$250k sesgan la media hacia arriba.
# La mediana de $32k representa mejor el ingreso típico y es robusta para el modelo lineal.
# DECISIÓN DOC: Ver celda Markdown 3.2 arriba. Alternativas como KNN descartadas por costo/beneício.
# SUPUESTO: Nulos son MAR. Validar con negocio si hay sesgo de reporte en altos ingresos.
mediana_ingreso = df['ingreso_mensual'].median()
df['ingreso_mensual'] = df['ingreso_mensual'].fillna(mediana_ingreso)

# Guardamos el valor para reproducibilidad en producción
with open('artifacts/imputacion_params.json', 'w') as f:
    json.dump({'ingreso_mensual_mediana': mediana_ingreso}, f)
```

### **¿Por qué importa esta diferencia para el equipo?**

1. **Onboarding**: Un nuevo DS lee tu notebook y entiende en 5 min si puede cambiar mediana por media sin romper supuestos.
2. **Code review**: Tu lead puede cuestionar "¿por qué no MICE?" y tú ya respondiste en la doc.
3. **Debugging en producción**: El modelo cae 10 puntos. Revisas la doc y ves que el supuesto MAR ya no se cumple porque cambió el formulario de captura. Ya sabes dónde mirar.
4. **Auditoría**: Regulator pregunta por qué imputaste así para el modelo de riesgo crediticio. Tienes la trazabilidad completa.

**Framework rápido para documentar cualquier decisión analítica**:

> **Decidimos en lugar de porque [razón basada en datos], asumiendo. El riesgo es.**[X][Y][supuesto][Z]

Si tu documentación no pasa ese template, solo estás documentando el *qué*.

## 3. EDA en la Industria: ¿Cómo se estructura un Análisis Exploratorio de Datos (EDA) en un equipo profesional de Data Science? ¿Qué entregables se esperan al finalizar el EDA antes de pasar a la fase de modelado, y qué herramientas o formatos (como reportes, dashboards o notebooks) se utilizan comúnmente para comunicar los hallazgos a stakeholders no técnicos?

En la industria, el EDA no es “hacer unos plots y ya”. Es la fase donde defines si el problema tiene solución con los datos que tienes. Si el EDA está mal, el modelado es tirar tiempo.

### **¿Cómo se estructura un EDA en un equipo profesional?**

Un EDA pro no es lineal. Es iterativo y tiene 5 capas, de arriba hacia abajo:

| Etapa | Objetivo | Preguntas que responde | Output interno |
| --- | --- | --- | --- |
| **1. Definición del problema** | Alinear con negocio antes de tocar datos | ¿Qué métrica de negocio movemos? ¿Qué decisión se tomará con el modelo? | 1-pager con hipótesis |
| **2. Sanity check de datos** | Validar que los datos existen y tienen sentido | ¿Tenemos la variable target? ¿Faltan 80% de filas? ¿Los tipos son correctos? | `pandas-profiling` + tabla de coverage |
| **3. Análisis univariado** | Entender cada variable sola | Distribuciones, outliers, nulos, cardinalidad. ¿`edad` tiene valores de 200 años? | Histogramas, boxplots, tablas |
| **4. Análisis bivariado/multivariado** | Entender relaciones con target y entre features | ¿Qué variables correlacionan con churn? ¿Hay data leakage? ¿Multicolinealidad? | Heatmaps, scatter plots, tests estadísticos |
| **5. Validación de hipótesis** | Probar supuestos de negocio con datos | “Los usuarios nuevos cancelan más”: ¿cierto? ¿En qué segmento? | Gráficas + prueba chi-cuadrado/t-test |

**Regla clave**: Cada etapa termina con una decisión go/no-go. Si en etapa 2 ves que el target tiene 99% de una clase y no hay forma de conseguir más datos, paras ahí. No quemas 2 semanas haciendo plots bonitos.

### **Entregables esperados antes de modelar**

Antes de escribir `from sklearn import`, el equipo espera estos 3 entregables firmados:

**1. EDA Técnico - Notebook de referencia**  
Dueño: Data Scientist.  
Contiene: Código reproducible, todas las gráficas, decisiones de limpieza, diccionario de datos.  
Formato: `.ipynb` en repo + ejecutado de arriba a abajo. En Colab: “Run all” sin errores.  
Criterio de aceptación: Otro DS puede tomar el notebook y llegar a los mismos insights.

**2. Data Quality Report - Para Data Engineer/Product**  
Dueño: DS + DE.  
Contiene: % nulos por columna, outliers detectados, inconsistencias, reglas de negocio violadas.  
Formato: Tabla en Confluence/Notion o slide. Columnas: `Variable | Problema | % Afectado | Acción propuesta | Dueño`.  
Ejemplo: `| fecha_compra | 12% fechas futuras | 12% | Filtrar + alertar a pipeline | @data_eng |`

**3. Insights Deck - Para stakeholders no técnicos**  
Dueño: DS que presenta.  
Contiene: 5-7 hallazgos clave que impactan el modelado o el negocio. Cero jerga técnica.  
Formato: 5 slides máximo. Cada slide = 1 insight + 1 gráfica simple + “So what?”  
Criterio: Un PM debe poder explicar los hallazgos en reunión de directores sin ti.

**Bonus que separa juniors de seniors: Feature Plan**  
Tabla que dice qué variables entran al modelo v1, cuáles se transforman, y cuáles se descartan con razón.  
`| Variable | Acción | Justificación |`  
`| edad | Binning 18-25, 26-35... | Relación no lineal con target |`  
`| id_usuario | Drop | Alto cardinal, es leakage |`

### **Herramientas y formatos para comunicar a no técnicos**

A stakeholders no les importa tu `seaborn`. Les importa riesgo, dinero y decisiones. Usa esto:

| Audiencia | Herramienta/Formato | Por qué funciona | Qué evitar |
| --- | --- | --- | --- |
| **PMs / Product** | Slides con 1 gráfica gigante + titular de impacto | Van directo al “qué hacemos con esto” | Matrices de correlación 20x20 |
| **Directores / C-Level** | 1-page memo o dashboard en Looker/Tableau | Resumen ejecutivo: 3 bullets, 1 gráfica, 1 recomendación | Jerga: “p-value”, “AUC”, “skewness” |
| **Negocio / Ops** | Dashboard interactivo con filtros | Ellos exploran su segmento: “muéstrame solo CDMX” | Notebooks crudos de Jupyter |
| **Legal/Compliance** | PDF con metodología + data lineage | Necesitan trazabilidad para auditoría | Links a Colab que se rompen |

**Stack común en equipos top:**

1. **Exploración**: Jupyter/Colab + `ydata-profiling`, `sweetviz` para EDA automático inicial.
2. **Visualización**: `plotly` para interactividad. Si el stakeholder tocará datos, mejor un dashboard.
3. **Comunicación**: `Quarto` o `papermill` para generar reportes HTML/PDF desde el notebook. El código se oculta, quedan solo insights.
4. **Dashboarding**: Looker Studio, Tableau, Streamlit para self-service. PowerBI si es cliente enterprise.
5. **Documentación**: Notion/Confluence con el “EDA Summary”. Link al notebook + al dashboard + 3 decisiones clave.

### **Ejemplo: Slide para stakeholder no técnico**

**Mal**:  
Título: “Distribución de `tenure_months` con test Kolmogorov-Smirnov”  
Gráfica: Histograma con 30 bins + curva KDE  
Conclusión: “p < 0.05, rechazamos normalidad”

**Bien**:  
Título: “60% de los clientes que cancelan tienen menos de 3 meses con nosotros”  
Gráfica: Barras simples: 0-3 meses vs 3+ meses, con % de churn encima  
Conclusión: “Oportunidad: Mejorar onboarding. Siguiente paso: modelo enfocado en clientes nuevos”

**La prueba ácida**: Si borras los ejes y el título técnico, ¿el stakeholder sigue entendiendo el mensaje? Si sí, lo lograste.

**Anti-patrón que mata equipos**: “EDA infinito”. Pon time-box: 20% del tiempo del proyecto máximo. Al día 5 debes tener el Insights Deck, aunque falten plots. El modelo v1 te dará más preguntas para el EDA v2.
